# Tag Glossary Template

## Purpose

This document defines the approved tags for your knowledge vault. Using a controlled vocabulary ensures consistency and makes searching effective.

**Rules:**
1. Before creating a new tag, check if an existing tag works
2. All tags should be lowercase with hyphens (no spaces)
3. Use singular forms (client not clients)
4. Add new tags to this glossary before using them
5. Review and consolidate tags quarterly

## How to Use This Glossary

### Adding a New Tag

Before adding a new tag, ask:

1. **Does an existing tag cover this?** Check the glossary first.
2. **Will I use this tag more than once?** One-off tags add noise.
3. **Is it specific enough?** "Work" is too broad; "client-work" is better.
4. **Is it too specific?** "dataflow-january-meeting" is too narrow; "dataflow" + "meetings" is better.

If you genuinely need a new tag:

1. Add it to the appropriate category above
2. Document when to use it
3. Give an example

### Tag Naming Conventions

| Do | Don't |
|----|-------|
| `client-work` | `Client Work` |
| `meeting` | `meetings` |
| `q1-planning` | `Q1 Planning` |
| `status-update` | `StatusUpdate` |

### How Many Tags Per Document?

Aim for 2-5 tags per document.

- **Too few (1)**: Not specific enough to be useful
- **Just right (3-4)**: Captures what + who + when
- **Too many (7+)**: Dilutes meaning, hard to maintain

### Example Tag Combinations

**Client meeting notes:**
```yaml
tags:
  - meetings
  - client-work
  - dataflow
```

**Internal weekly update:**
```yaml
tags:
  - reports
  - weekly
  - internal
```

**Urgent proposal:**
```yaml
tags:
  - client-work
  - urgent
  - needs-review
```

## Quarterly Tag Audit

Every quarter, review your tags:

1. **Unused tags**: Remove any not used in 3+ months
2. **Similar tags**: Consolidate duplicates (meeting vs meetings)
3. **Over-used tags**: If a tag is on 80%+ of documents, it's not useful
4. **Missing tags**: Add any you've been manually adding repeatedly
